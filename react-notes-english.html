useState Hook
The way to store data in React
What is it?
Simple Definition

useState is a React Hook that gives us the power to store data (variables) inside a component — and whenever that data changes, the screen automatically updates!


→ Change a normal JavaScript variable → the screen does nothing.

→ Change a useState variable → React re-renders the screen.

Think of useState as a Magic Board — write something on the board, and it automatically shows up on the wall (screen). Erase the board → the wall updates too!
Why do we use it?
Without useState
  • Data changes
  • Screen stays the same
  • User sees nothing
With useState
  • Data changes
  • React notices
  • Screen updates instantly!

Real cases: Counter, Login form, Dark mode toggle, Shopping cart

Real Life Example
Example

Imagine you want to build a like button (like on Instagram). When the user clicks, the likes increase. This is exactly what useState does!

→ Likes = state (data)

→ Button click = state change

→ Screen update = React magic

Syntax + Code
 Counter.jsx
import React, { useState } from 'react';

function Counter() {
  // Declare useState
  const [count, setCount] = useState(0);
  //        ↑      ↑            ↑
  //    variable  updater   initial value

  return (
    <div>
      <h2>Count: {count}</h2>
      <button onClick={() => setCount(count + 1)}>
        +1 Increase
      </button>
      <button onClick={() => setCount(0)}>
        Reset
      </button>
    </div>
  );
}

export default Counter;
Code Breakdown
import {`{ useState }`} from 'react'
Import the useState hook from the React library — this is required first!
const [count, setCount] = useState(0)
Array destructuring: count = current value, setCount = the function to change the value, 0 = starting value
{count}
Show the value of count inside curly braces in JSX
onClick={() => setCount(count + 1)}
Call setCount on button click, add 1 to count
setCount(0)
Set count back to 0 (reset)
Important Rule

You should NEVER change state directly: count = count + 1

Always use a setter function: setCount(count + 1)

Common Beginner Mistakes
Writing count = count + 1 directly
Always use setCount(count + 1). Direct modification doesn't let React know.
Writing useState inside an if/loop
Always write useState at the top of the component — that's what React's "Rules of Hooks" says.
const [count] = useState(0) — forgetting the setter
You need to write both: const [count, setCount]
Not importing first
import React, { useState } from 'react'; — this line must be written FIRST
Interview Questions
What is useState?
A React Hook that lets you manage state in a functional component. When state changes, the component re-renders.
What's the difference between state and a variable?
When a normal variable changes, the screen doesn't update. When state changes, React automatically re-renders the component.
What can the initial value of useState be?
Anything — number, string, boolean, array, object, null. For example: useState([]), useState({}), useState(false)
Why don't we change state directly?
Because then React doesn't re-render. Calling the setter function signals React that the screen needs to update.

Quick Revision — useState

  • useState = the hook for storing state
  • Syntax: const [value, setValue] = useState(initialValue)
  • Change state → React updates the screen
  • NEVER mutate directly, always use the setter
  • Write it at the top of the component, never inside a loop/if
Topic 02 of 8
useEffect Hook
The way to handle side effects
What is it?
Simple Definition

useEffect is a hook that tells React: "Do this task — AFTER rendering"

Side effects means: making an API call, setting a timer, adding an event listener, changing the document title — basically, work outside of React

useEffect = "the bodyguard that runs after rendering" — it handles the behind-the-scenes work for React (API, timers, DOM changes)
Real Life Example
Example

Imagine a page that needs to show news. After the page loads, news needs to be fetched from the server.

→ Page rendered

→ useEffect ran

→ News came from the API

→ setNews was called → screen updates!

Syntax — 3 Forms
 useEffect — 3 tarike
import { useEffect, useState } from 'react';

// ── FORM 1: Runs after every render ──
useEffect(() => {
  console.log('I run on every render!');
}); // no dependency array

// ── FORM 2: Only once (on mount) ──
useEffect(() => {
  console.log('Only once — on page load!');
  // Make the API call here
}, []); // empty array = only on mount

// ── FORM 3: On specific value change ──
useEffect(() => {
  console.log('count changed!');
}, [count]); // runs only when count changes
 Full Example — API Fetch
function NewsPage() {
  const [news, setNews] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // This will run after the page loads
    fetch('https://api.example.com/news')
      .then(res => res.json())
      .then(data => {
        setNews(data);
        setLoading(false);
      });
  }, []); // [] = only once

  if (loading) return <p>Loading...</p>;

  return (
    <ul>
      {news.map(item => (
        <li key={item.id}>{item.title}</li>
      ))}
    </ul>
  );
}
Code Breakdown
useEffect(() => { ... }, [])
Run only once after the component appears on screen (like componentDidMount in classes)
fetch(...).then(...)
Request data from the internet, handle it with a promise chain
setNews(data)
Put the data received from the API into state → screen updates
setLoading(false)
Turn off the loading state → remove the "Loading..." message
Cleanup Function

Some effects need to be shut down (timers, event listeners):

 Cleanup
useEffect(() => {
  const timer = setInterval(() => {
    console.log('tick');
  }, 1000);

  return () => clearInterval(timer); // cleanup
}, []);
Common Mistakes
Forgetting the dependency array → Infinite loop!
Always write [] or [variable] — otherwise it'll run on every render
Writing an async function directly inside useEffect
Create a separate async function inside and call it
Forgetting cleanup (memory leak!)
Always write return () => cleanup() for timers and subscriptions
Interview Questions
Why do we use useEffect?
To handle side effects — API calls, timers, subscriptions — work that's outside React's rendering.
What's the difference between [] and [count]?
[] = runs only once (on mount). [count] = runs only when the value of count changes.
When is a cleanup function necessary?
When using setInterval, setTimeout, an event listener, or a subscription — they must be cleared on unmount, otherwise there's a memory leak.

Quick Revision — useEffect

  • useEffect = run side effects after rendering
  • [] = only once | [x] = when x changes | no array = every time
  • API calls, timers, listeners = useEffect's job
  • Cleanup: return () => { ... } — avoid memory leaks
  • Don't write an async function directly, create one inside

Props & PropTypes
The way to send data to components
What is it?
Simple Definition

Props (Properties) = the way to send data from a parent component to a child component.

Just like you pass arguments to a function — that's how you pass props to a component!

→ Props always flow Parent → Child (one-way)

→ Child cannot change props (read-only)

Props = a Tiffin Box. Mom (parent) sends food, the child eats it. The child can't change the food!
Syntax + Code
 Props Example
// ── Child Component ──
function WelcomeCard({ name, age, city }) {
  return (
    <div className="card">
      <h2>Hello, {name}! </h2>
      <p>Age: {age} years</p>
      <p>City: {city}</p>
    </div>
  );
}

// ── Parent Component ──
function App() {
  return (
    <div>
      <WelcomeCard
        name="Rahul"
        age={22}
        city="Delhi"
      />
      <WelcomeCard
        name="Priya"
        age={19}
        city="Mumbai"
      />
    </div>
  );
}
Code Breakdown
function WelcomeCard({ name, age, city })
Destructure props and use them directly — cleaner code
name="Rahul"
String props — write them in quotes
age={22}
Number props — write them in curly braces
{name} JSX mein
Display the prop's value using curly braces
PropTypes — Type Safety
What is PropTypes?

PropTypes ensures the right type of data reaches the component. If the wrong type is passed, a warning shows up in the console!

 PropTypes Example
import PropTypes from 'prop-types';

function WelcomeCard({ name, age, city }) {
  return <h2>{name} - {age} - {city}</h2>;
}

// Type validation
WelcomeCard.propTypes = {
  name: PropTypes.string.isRequired,  // string + required
  age:  PropTypes.number,             // number (optional)
  city: PropTypes.string,             // string (optional)
};

// Default values
WelcomeCard.defaultProps = {
  city: 'India',
  age:  18,
};
Common Mistakes
Using props.name directly without destructuring
Writing { name, age } in the function parameter is cleaner
Changing props in the child (props.name = "new")
Props are read-only! Use state if you need to change something
Passing a number in quotes: age="22"
Pass numbers in curly braces: age={22}
Interview Questions
What's the difference between Props and State?
Props come from the parent and are read-only. State lives inside the component and can change.
Why are props one-way?
Data flow stays predictable — parent to child only. This makes debugging easier.
Why do we use PropTypes?
For runtime type checking — if the wrong type is passed, you get a warning, so bugs are found faster.

Quick Revision — Props

  • Props = data from Parent to Child
  • One-way flow: Parent → Child only
  • Read-only — child cannot change it
  • String = "value", Number/Boolean = {value}
  • PropTypes = type safety + defaultProps = backup values

Component Lifecycle
The phases of a component's life
What is it?
Simple Definition

Every React component has a "life" — appearing on screen, updating, and leaving the screen. These are called Lifecycle Phases.

3 Phases
1
Mount → the component appears on screen for the first time
2
Update → state or props change
3
Unmount → the component is removed from the screen
Lifecycle = a person's life: Being born (Mount) → Growing up/Changing (Update) → Dying (Unmount)
Functional Components mein Lifecycle

useEffect is used to manage lifecycle in functional components:

 Lifecycle with useEffect
import { useEffect, useState } from 'react';

function LifecycleDemo() {
  const [count, setCount] = useState(0);

  // ── 1. MOUNT: Only once, the first time ──
  useEffect(() => {
    console.log('Component mounted');

    // ── 3. UNMOUNT: On the way out ──
    return () => {
      console.log('Component unmounted');
    };
  }, []); // empty array = only on mount/unmount

  // ── 2. UPDATE: when count changes ──
  useEffect(() => {
    console.log('count updated', count);
  }, [count]);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increase
      </button>
    </div>
  );
}
Class vs Functional Comparison
Class Component (Purana)
  • componentDidMount
  • componentDidUpdate
  • componentWillUnmount
Functional (Naya)
  • useEffect(fn, [])
  • useEffect(fn, [dep])
  • useEffect(() => return fn, [])
Interview Questions
How many phases does the React lifecycle have?
3: Mount (appearing), Update (changing), Unmount (leaving).
What's the functional equivalent of componentDidMount?
useEffect(() => { ... }, []) — with an empty dependency array.
When does the cleanup function run?
Before the component unmounts — inside useEffect's return function.

Quick Revision — Lifecycle

  • Mount → useEffect(fn, [])
  • Update → useEffect(fn, [dep])
  • Unmount → return function inside useEffect
  • Class components had separate methods, now useEffect is enough

useContext Hook
The way to share global data
What is it?
Simple Definition

useContext is a hook that lets you use "global data" directly in any component — without props drilling!

→ Problem: needing to send a piece of data deep down through components

→ Solution: store it in Context, anyone can use it

Context = a WhatsApp Group. The Admin (Provider) sends a message, all members (Consumer) read it directly — no need to forward it through people in between!
Real Life — Props Drilling Problem
Props Drilling (Problem)

App → Header → Navbar → UserAvatar

Even if only UserAvatar needs the user's name, it still has to be passed through all three! Very tedious!

useContext (Solution)

Store the name in UserContext → UserAvatar can grab it directly!

Syntax + Code
 useContext — Step by Step
import { createContext, useContext, useState } from 'react';

// ── STEP 1: Create the context ──
const UserContext = createContext(null);

// ── STEP 2: Wrap with Provider ──
function App() {
  const [user, setUser] = useState({ name: 'Rahul', role: 'Admin' });

  return (
    <UserContext.Provider value={user}>
      <Header />
      <Dashboard />
    </UserContext.Provider>
  );
}

// ── STEP 3: Use in any child ──
function UserAvatar() {
  const user = useContext(UserContext);
  // No props needed! Direct access 

  return <div>Hello, {user.name}! ({user.role})</div>;
}
Code Breakdown
createContext(null)
Create a new context object, null = default value
<UserContext.Provider value={user}>
Provider gives data to all children — whatever you put in the value prop, everyone gets it
useContext(UserContext)
Get the context's value directly in the child — no prop was passed!
Best Use Cases
  • Theme (dark/light mode)
  • User authentication data
  • Language/locale settings
  • Shopping cart
Common Mistakes
Writing createContext inside a component
Always write it outside the component, at the top of the file
Using context outside the Provider
Wrap the component inside the Provider — only then will it get the value
Putting all state into context
Only use it for truly global data — keep local state in useState
Interview Questions
Why do we use the Context API?
To avoid props drilling — when data needs to be sent to deeply nested components.
What's the difference between Context and Redux?
Context is for simple global state. Redux is for complex state management — time-travel debugging, middleware support.
What's the difference between createContext and useContext?
createContext creates the context (once). useContext lets you get the context's value in any child component.

Quick Revision — useContext

  • Solves the props drilling problem
  • 3 steps: createContext → Provider → useContext
  • Whatever is inside the Provider, that gets the value
  • Perfect for Theme, User, Language

React Router
Creating multi-page navigation
What is it?
Simple Definition

React Router is a library that lets you create multiple pages/routes in a React app — without a page reload!

→ /home → show the Home page

→ /about → show the About page

→ /profile/123 → show User 123's profile

React Router = a Metro Map. There are different stations (routes), and you can go straight to any station (page) without passing through all the others!
Installation
 Terminal
npm install react-router-dom
Syntax + Code
 App.jsx — Full Router Setup
import {
  BrowserRouter,
  Routes,
  Route,
  Link,
  useParams,
  useNavigate
} from 'react-router-dom';

// ── Pages ──
function Home()    { return <h1>Home page</h1>; }
function About()   { return <h1>ℹ About Us</h1>; }
function NotFound(){ return <h1> 404 — Page not found!</h1>; }

function UserProfile() {
  const { id } = useParams(); // Extract id from URL
  return <h1>User #{id}'s profile</h1>;
}

// ── Main App ──
function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Home</Link> |
        <Link to="/about">About</Link>
      </nav>

      <Routes>
        <Route path="/"           element={<Home />} />
        <Route path="/about"     element={<About />} />
        <Route path="/user/:id" element={<UserProfile />} />
        <Route path="*"          element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  );
}
Code Breakdown
<BrowserRouter>
Wrap the whole app in this — the routing container
<Routes>
All routes are written here — only one matches
path="/" element={<Home />}
If the URL is / then show the Home component
path="/user/:id"
:id = dynamic parameter — can be any number/string
useParams()
Extract the dynamic value from the URL (like { id: "42" })
<Link to="/about">
Use this instead of a normal <a> tag — no page reload!
path="*"
If no route matches, show this 404 page
useNavigate — Programmatic Navigation
 useNavigate
function LoginPage() {
  const navigate = useNavigate();

  function handleLogin() {
    // Send to home after login success
    navigate('/');
  }

  return <button onClick={handleLogin}>Login</button>;
}
Common Mistakes
Using <a href="/about"> instead of Link
Use <Link to="/about"> — <a> causes a page reload!
Using useNavigate outside BrowserRouter
The whole app should be inside BrowserRouter
Writing path="*" first among routes
Always write the 404 route at the very bottom
Interview Questions
What does React Router do?
Handles multiple pages/views in a Single Page Application — the URL changes but the page doesn't reload.
What's the difference between Link and the <a> tag?
Link is a React Router component that does client-side navigation. The <a> tag causes a full page reload.
How do you create dynamic routes?
Write path="/user/:id" and extract the id with useParams().

Quick Revision — React Router

  • BrowserRouter → Routes → Route hierarchy
  • Use <Link>, not <a>
  • :id = dynamic parameter, extract with useParams()
  • useNavigate = navigation from code
  • path="*" = 404 page, at the very bottom

useRef Hook
DOM access + storing values without re-render
What is it?
Simple Definition

useRef has 2 main jobs:

1
Directly accessing a DOM element (like focusing an input)
2
Storing a value without a re-render (like a timer ID)
useRef = a Sticky Note — write any value and stick it on. Changing the note doesn't repaint the room (screen)!
Use Case 1: DOM Access
 Auto-focus input
import { useRef, useEffect } from 'react';

function SearchBar() {
  const inputRef = useRef(null);

  useEffect(() => {
    // Focus the input as soon as the page loads
    inputRef.current.focus();
  }, []);

  return (
    <input
      ref={inputRef}        // attach ref to the input
      placeholder="Search..."
    />
  );
}
Use Case 2: Value Store (No Re-render)
 Timer with useRef
import { useRef, useState } from 'react';

function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  const timerRef = useRef(null); // store the timer ID

  function start() {
    timerRef.current = setInterval(() => {
      setSeconds(s => s + 1);
    }, 1000);
  }

  function stop() {
    clearInterval(timerRef.current); // stop the timer
  }

  return (
    <div>
      <p>{seconds} seconds</p>
      <button onClick={start}>▶ Start</button>
      <button onClick={stop}>⏹ Stop</button>
    </div>
  );
}
Code Breakdown
const inputRef = useRef(null)
Create a ref, null = no DOM element initially
ref={inputRef}
Attach the ref to the input element — now inputRef.current = the input element
inputRef.current.focus()
Call DOM methods directly — focus, blur, click, all possible
timerRef.current = setInterval(...)
Store the timer ID in the ref — changing it won't trigger a re-render
useState vs useRef

useState
Change → Re-render
For UI values

useRef
Change → NO Re-render
DOM access + side values

Common Mistakes
Using useRef to trigger a re-render
If you need a re-render, use useState — useRef doesn't update the screen
Accessing ref.current before render
Access it in useEffect or an event handler — the DOM is only ready after render
Interview Questions
What's the difference between useRef and useState?
useState triggers a re-render on change. useRef does NOT trigger a re-render on change. useRef is also used for DOM access.
What is .current in useRef?
This holds the actual stored value — whether it's a DOM element or a variable. React manages this directly.

Quick Revision — useRef

  • DOM element access: ref attribute + inputRef.current.focus()
  • Value store: no re-render, the value lives in .current
  • Perfect for storing timer IDs, previous values
  • useState = need a screen update | useRef = just need the value

Custom Hooks
Creating your own custom hook
What is it?
Simple Definition

Custom Hook = creating your own hook that contains reusable logic.

→ When the same logic is used in many components

→ Pull it out into a custom hook

→ Use it from anywhere!

Rule: A custom hook's name must always start with "use"! (useCounter, useFetch, useLocalStorage)

Custom Hook = your own Tool. Just like a carpenter makes their own custom tool that comes in handy again and again — you make your own hook the same way!
Real Life Example
Problem without Custom Hook

If 5 different components need to fetch data from an API, you'd have to write the same useEffect + useState 5 times. So much repetition!

Solution: create a custom hook called useFetch → write the logic just once

Example 1 — useCounter
 useCounter.js
import { useState } from 'react';

// Custom Hook — name must start with "use"!
function useCounter(initialValue = 0) {
  const [count, setCount] = useState(initialValue);

  const increment = () => setCount(count + 1);
  const decrement = () => setCount(count - 1);
  const reset     = () => setCount(initialValue);

  return { count, increment, decrement, reset };
}

// ── Use in any component ──
function ShoppingCart() {
  const { count, increment, decrement } = useCounter(0);

  return (
    <div>
      <button onClick={decrement}>-</button>
      <span>{count} items</span>
      <button onClick={increment}>+</button>
    </div>
  );
}
Example 2 — useFetch (Real World)
 useFetch.js
import { useState, useEffect } from 'react';

function useFetch(url) {
  const [data, setData]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    fetch(url)
      .then(res => res.json())
      .then(d  => { setData(d);    setLoading(false); })
      .catch(e => { setError(e);   setLoading(false); });
  }, [url]);

  return { data, loading, error };
}

// ── Use it — so clean! ──
function UserList() {
  const { data, loading, error } = useFetch(
    'https://jsonplaceholder.typicode.com/users'
  );

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error aaya!</p>;

  return (
    <ul>
      {data.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
Code Breakdown
function useFetch(url)
Starts with "use" — this is a custom hook. Takes a url parameter
return { data, loading, error }
Return an object — the caller can destructure and use it
const { data, loading } = useFetch(url)
Call the custom hook like a normal function — magic!
Custom Hook Rules
  • Name should always start with "use"
  • You can use other hooks inside
  • Only call it inside functional components
  • Don't call it inside a loop/if
Common Mistakes
Not starting the name with "use" (like myHook)
Always write useMyHook — so React knows this is a hook
Calling a custom hook inside a regular function
Only call it inside functional components or other hooks
Creating a custom hook for everything
Only create one when the same logic repeats in 2+ places
Interview Questions
What is a custom hook?
Your own hook that contains reusable logic. Its name starts with "use" and it can use other hooks.
What's the difference between a custom hook and a normal function?
React hooks (useState, useEffect) can be used inside a custom hook. Not inside a normal function.
When should you create custom hooks?
When the same logic repeats in 2 or more components — the DRY principle (Don't Repeat Yourself).
Does a custom hook share state?
No! Each component keeps its own separate state. The hook only shares logic, not state.

Quick Revision — Custom Hooks

  • Start the name with "use" — a required rule!
  • Reusable logic = Custom Hook
  • useState, useEffect, everything can be used inside
  • State isn't shared — only logic is shared
  • Follow the DRY principle — write once, use repeatedly
